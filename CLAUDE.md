# CLAUDE.md

このファイルは、このリポジトリで作業する Claude Code(および開発者)向けのガイドです。

## プロジェクト概要

**KURA:S** ― 家庭内の在庫(食材・日用品)を管理するモバイル向けWebアプリ(PWA的なSPA)。
AI写真判定・手動入力で在庫を登録/消費でき、在庫が少なくなると買い物リストに自動連携する。

- 認証・DB: Supabase
- AI商品判定: Google Gemini API(`gemini-3.1-flash-lite`)
- ビルドツールなし。素のHTML/CSS/JS(ESモジュール)のみで構成された静的サイト

## 開発ルール

### 基本方針

- **保守性・拡張性を最優先**する。奇抜な最適化やトリッキーな書き方より、誰が読んでも分かるコードを優先する
- **既存機能・既存UIを壊さない**。動作確認が難しい変更は、影響範囲を明示したうえで慎重に進める
- **必要最小限のファイルのみ編集**する。関係のないファイルには触れない
- **変更範囲を最小限にする**。ついでの修正・ついでの整形はしない
- **リファクタリング時は見た目・動作を一切変更しない**。見た目や挙動が変わる場合はリファクタリングと呼ばない
- **実装前に必ず設計案を提示**し、私の承認後に実装する
- **大規模な変更は小さなステップに分けて**進める。1つのステップごとに動作確認できる単位にする
- **実装後は必ずローカルで動作確認**し、結果を報告する

### 編集プロセス(必ず守ること)

1. **実装前に、変更方針・変更対象ファイル・実行予定コマンドを含む設計案を提示する**
2. **私(ユーザー)の承認を得る**
3. **承認後は、設計案の範囲内であれば、個々の編集・コマンド実行のたびに確認を求めず、まとめて進める**
4. **不要なリネームを行わない**(ファイル名・変数名・関数名など)
5. **不要なフォーマット変更を行わない**(インデント統一、改行整形、クォート統一など、指示されていない整形はしない)

設計案の範囲を超える変更が必要になった場合(対象外ファイルの編集、方針と異なるアプローチへの変更など)は、あらためて確認を取る。

### AI実装

- AI呼び出し回数を最小限にする
- キャッシュを最優先で利用する
- 商品マスタを優先利用する
- 将来の拡張性を考慮した設計にする
- 商品マスタ未ヒット時にAIへ標準商品名を判定させる際は、既存の標準商品名・検索キーワードとの文字列類似度(bigram重なり係数、`js/productMaster.js` の `fetchKnownCanonicalNameCandidates`)でローカルに絞り込んだ候補をヒントとして渡し、表記ゆれ(例:「豚小間切れ」⇔「豚小間切れ肉」)で同じ標準商品名を再利用しやすくする。この絞り込みはAIを呼ばずに行うため、AI呼び出し回数は増えない
- 使用モデルは `gemini-3.1-flash-lite` に固定している(2026-07-16)。無料枠の1日あたりリクエスト上限が少ない `gemini-flash-latest`(Gemini 3.5 Flash系、無料枠20回/日)で上限超過エラーが頻発したため切替。別モデルへの自動フォールバックは実装しない方針
- 標準商品名・サブカテゴリー・検索キーワードのひらがな読みは、商品属性生成の同じAI呼び出し(`identifyProductAttributes`)にまとめて含めており、専用の追加呼び出しは行わない。カテゴリーはenumが固定のためAIを使わず`js/productMaster.js`の固定表(`getCategoryReading`)で読みを持つ
- 商品マスタ解決(`resolveProductMaster`)は、入力文字列の正規化結果(`normalizeProductName`)が`product_name_alias`(過去に入力した生の商品名のキャッシュ)に無くても、続けて`product_master.canonical_normalized_name`(既存の標準商品名そのもの)と完全一致するかを確認し、一致すればAIを呼ばずその商品マスタを再利用する(該当raw_nameは以後のためにaliasへもキャッシュする)。一致するのは文字列が完全に同じ場合のみで、bigram類似度のような曖昧な一致は使わない(`forceRegenerate: true`のときはこのチェックもスキップし必ずAIを呼ぶ)

## UI/UX方針

- デザインコンセプトは **「PayPayの使いやすさ」× 「ME:Iの未来感・高級感」**
  - PayPay的: タップ数が少ない、迷わない、目的の操作に最短で辿り着ける導線
  - ME:I的: 上品なパープル基調、繊細な余白とタイポグラフィ、控えめで上質なアニメーション
- **シンプルで、情報量が多くても見やすいUI**にする(在庫一覧・買い物リストなど情報密度が上がりやすい画面で特に意識する)
- **スマホファースト**(`.wrap` の `max-width: 460px` を前提に設計。PC幅対応は二の次)
- **主な利用環境は iPhone 14 相当**(論理解像度 390×844pt、@3x)。UI変更時はこのビューポートで折り返し・見切れが起きないか確認する(横並びの列を増やしすぎない、ボタン内テキストが折り返さない、など)
- **アクセシビリティ**を意識する(`aria-label`、タップ領域の大きさ、コントラスト、フォーカス可視性など)

## コード方針

### CSS(`css/`)
- **画面・コンポーネント単位で管理**する(例: `home.css` はホーム画面、`modal.css` はモーダル共通部品)
- 共通のトークン(色・角丸・影など)は `common.css` の CSS変数(`:root`)にまとめる。新しい色や数値を各ファイルにベタ書きしない
- 新しい画面・コンポーネントを追加する場合は、既存の分割方針に合わせて新規CSSファイルを作り、`index.html` の `<head>` に `<link>` を追加する

### JavaScript(`js/`)
- **機能単位でモジュールを分割する**(例: `consume.js` = 消費関連、`shopping.js` = 買い物リスト関連)。既存の分割粒度に合わせる
- **共通処理は再利用できるようにまとめる**(`utils.js`, `elements.js` などに集約。同じロジックをコピペしない)
- 新規モジュールは `js/main.js` に副作用importとして追加する(既存の初期化パターンに合わせる)
- モジュール間の依存は `export` / `import` を使う(グローバル変数は最小限に)。ただし `onclick="..."` 属性から呼ぶ必要がある関数は `window.xxx` を使う既存パターンを踏襲する
- **コメントは必要最小限**。「何をしているか」ではなく「なぜそうしているか」が非自明な場合のみ書く
- **読みやすいコード**を優先する(早すぎる抽象化・過剰な汎用化はしない)

## データモデル(在庫まわり)

在庫は3層に分かれている。

- **`items`** = 商品(名前・分類・単位・`product_master_id`)。数量・賞味期限は持たない(`quantity`/`expiry_date`列は`item_lots`への移行完了により削除済み)。`low_stock_threshold`列は残っているが、商品マスタが無いフォールバック商品(単独カード)でのみ使う(商品マスタがある商品は`product_master.low_stock_threshold`を使うため、この列は使われない)
- **`item_lots`** = 在庫ロット(数量・賞味期限・購入日・価格)。1商品(`items`)に対して複数持てる。**「同じ商品 かつ 同じ賞味期限」のときだけ既存ロットに数量を加算し、賞味期限が異なれば必ず新規ロットになる**。ロットの取得・表示・消費順は常に `expiry_date ASC NULLS LAST`(賞味期限が近い順、未設定は最後)。購入日(`purchase_date`)は在庫確認画面には出さず、各商品名の「ⓘ」インフォメーションシート(画面構成10)でのみ確認できる。価格(`price`、nullable)は手動登録画面でのみ入力でき(食費・日用品費の計算確認が目的)、**常に「1つ当たりの単価」(定量系は100gなど基準量あたりの単価)を保存する**(合計金額で入力した場合も登録時に単価へ変換する。消費が進むと`quantity`が減っていくため、単価を固定しておかないと後から消費した分の金額を計算できないため)。既存ロットへ数量を加算する際は新しい価格の入力が無ければ元の単価を保持する
- **`product_master`** / **`product_name_alias`** = 商品属性(標準商品名・カテゴリー・サブカテゴリー・保存場所・用途・検索キーワードなど、AIが生成)。ユーザー・世帯をまたいで共有されるマスタで、在庫のライフサイクル(登録・消費・削除)とは独立している。標準商品名(`canonical_name`)・サブカテゴリー(`sub_category`)・検索キーワード(`search_keywords`)にはひらがな読み列(`canonical_name_reading`・`sub_category_reading`・`search_keywords_reading`)があり、AI属性生成時に同じ呼び出しでまとめて生成される(カテゴリーのひらがな読みは固定enumのため、AIを使わず`js/productMaster.js`の`getCategoryReading`の固定表で持つ)。標準商品名(`canonical_name`、表示用)は商品マスタ 詳細/編集シートから編集できるが、`canonical_normalized_name`(AIが将来同じ標準商品名と判定した際に既存マスタを再利用するための照合キー)は編集の影響を受けない。`low_stock_threshold`は標準商品名(カード)単位の最低数量で、同じ標準商品名を共有するすべての商品名(`items`)に対して1つだけ持つ
- **`item_history`** = 購入・消費の追記専用ログ(ホーム画面の「購入履歴」ページで表示)。`item_lots`は消費で数量が0になると行ごと削除され、購入時も既存ロットへ数量を加算する場合があるため、あくまで「今の在庫状態」であって履歴ではない。数量が増減するたびに`js/items.js`の`logItemHistory()`が1行追記する(日付`occurred_at`・商品名`item_name`・標準商品名`canonical_name`・種別(食品/日用品)`item_type`・区分`event_type`(purchase/consumption)・数量・単位・価格は、いずれも変更時点のスナップショット。`item_type`は`js/items.js`の`effectiveType()`と同じ判定=商品マスタがあればその`type`、無ければ`items.category`)。`item_id`/`product_master_id`は`on delete set null`にしており、商品や商品マスタを削除しても履歴の行自体は残る(スナップショット列で読める)

在庫確認画面は、`items.product_master_id` が同じ商品を「標準商品名」(`product_master.canonical_name`)の1カードにまとめ、その下に実際の商品名(`items.name`)ごとの区画を表示する(商品マスタが無い商品は、従来どおり単一カードで表示するフォールバックになる)。カテゴリー見出しは `product_master.category`(無ければ`items.category`)、サブカテゴリー見出しは `product_master.sub_category`(無ければ「未分類」)を使う。

数量の増減・削除は必ず `item_lots.id` を主キーに操作する(ロットの数量が0になったらそのロットだけ削除し、他のロット・他の商品には影響させない)。買い物リストとの連動(在庫少なめによる自動追加/自動削除)は `js/shopping.js` の `syncShoppingListForItem()` に一本化されており、在庫の増減・消費・登録などすべての経路がこの関数を経由する。ロットの数量変更経路(`js/items.js`の`resolveLot()`・`persistLotQty()`)は、いずれも変更のたびに`logItemHistory()`で`item_history`へ増減分を記録する。`persistLotQty(lotId, itemId, newQty, previousQty)`は、呼び出し元(画面に表示中の値)から渡された`previousQty`との差分で増えれば購入・減れば消費として記録する(DBへの再取得に頼ると、その読み取りが何らかの理由で失敗した場合に増減方向を誤りかねないため、呼び出し元が既に把握している値をそのまま使う設計にしている)。

- **商品マスタがある商品**: 最低数量は標準商品名(カード)単位。同じ`product_master_id`を共有する**すべての商品名の在庫合計**と`product_master.low_stock_threshold`を比較する(`syncShoppingListForMaster()`)。判定式は`low_stock_threshold > 0 && combinedQuantity < low_stock_threshold`。個数系(個・パックなど)と定量系(g/ml/kg/L)の商品名が混在する場合、`js/quantity.js`の`computeCombinedStockQuantity()`が「定量系の合計÷定量系の商品数=平均値を個数系の数量に掛けて定量換算してから合算する」変換を行う(例: 200g,400g,2パック → 平均300g×2パック=600g換算、合計1200g)。定量系が1つも無ければ換算せずそのまま合算する。買い物リストへは標準商品名(`canonical_name`)で1件だけ追加し(`shopping_list.product_master_id`で重複判定)、複数の商品名が同時に低在庫になっても項目は増えない
- **商品マスタが無い商品(フォールバック)**: 従来通り、この商品(`items`)単独の在庫合計と`items.low_stock_threshold`を比較する(判定式は同じ`> 0 && < threshold`)。買い物リストへは`items.name`で追加する

最低数量が`0`の場合は、在庫が`0`でも自動追加しない(「管理しない」という意思表示として扱う、という運用上の意図は共通)。

新規商品登録時の `items.category` は、ユーザーが手入力するのではなく、商品マスタ解決(`resolveProductMaster`)が判定した `product_master.type`(食品/日用品)を自動的に反映する(`js/items.js` の `resolveItem`)。商品マスタの判定に失敗した場合のみ `product_master_id` は `null` のままとなり、登録画面に表示される「種別」フォールバック入力欄の値がそのまま `items.category` に入る。

## 画面構成

**共通通知欄**(`#app-notice`): ログイン後の`.app-header`(KURA:Sロゴ)と**同じカードの中**に置かれた通知行で、新しいカードは作らず`.app-header`自体の縦幅が伸びて表示される(`.app-header-top`がロゴ+ログアウト行、その下に`#app-notice`)。登録・消費・買い物リスト追加・商品属性の生成/再生成など、完了時の案内メッセージ(旧: 各パネル内に緑字で表示していたもの)はここに一本化して表示する(小さめの文字・表示後4秒で自動的に消える)。呼び出しは`js/utils.js`の`showAppNotice(text)`。商品登録時は「AIが商品属性を生成しました」/「既存の商品属性を利用しました」を先頭に付け、AIが新規に属性を作ったか既存を再利用したかを区別して表示する(`upsertItemByName`/`resolveItem`が返す`productMasterStatus`で判定)。入力エラーなど失敗時のメッセージは対象外で、従来どおり各パネル内に赤字で表示する。ログイン画面(`#auth-card`)はヘッダーが別(`.brand-header`)のため対象外。

1. **認証画面**(`#auth-card`): ログイン / 新規登録(Supabase Auth)
2. **ホーム**(`#view-home`):
   - 挨拶・日付表示
   - 「商品を登録」「商品を消費」の2枚のアコーディオンカード。開くとAI/手動の2ボタンが並ぶだけで、選ぶとどちらもその場では差し替わらず共通オーバーレイを開く(登録は`#register-overlay`・詳細は画面構成7、消費は`#consume-overlay`・詳細は画面構成8)。中身(AI判定結果・手動フォーム・検索・確定)の詳細はそれぞれの画面構成の項目を参照
   - 登録画面(AI・手動とも)に「分類」の手入力欄は無い。新規商品は商品マスタ(AI)が判定した`type`(食品/日用品)を`items.category`へ自動反映する。判定に失敗した場合のみ、その場に「種別」選択欄(食品/日用品)が表示され、選び直して再度「登録」を押すとAIを再呼び出しせずその値で登録される(個別登録・まとめて登録の両方に対応。まとめて登録では種別入力が必要な商品だけカードが残る)
   - 賞味期限のお知らせカード(現在はダミー表示)
   - 今後追加予定カード(AIレシピ・家族共有。破線の枠で「近日公開」と表示するプレースホルダー)。「購入履歴」カードのみ実装済みで、実線の枠でタップすると購入・消費履歴ページ(画面構成12)を開く
3. **買い物リスト**(`#view-shopping`): 買い物項目一覧のみで、画面上からの自由入力欄は無い(項目はホーム画面の「商品を登録」の手動/AIから追加するか、在庫少なめによる自動追加で増える)。「購入済み」ボタンを押すと商品登録オーバーレイ(画面構成7の「買い物リスト「購入済み」からの登録」)を開く(即座には確定しない)。各行には「削除」ボタンがある(編集機能は無い)
4. **在庫確認**(`#view-inventory`): タイトル下に「フィルター」(画面構成5)「並び替え」(画面構成6)ボタンを並べて設置。フィルターで種別/カテゴリー/サブカテゴリーを複数選択でき、その下には商品名検索欄(`#inventory-search`)があり、消費画面と同じ優先度スコアリング(`js/productMaster.js` の `computeItemSearchScore`。商品名>標準商品名>サブカテゴリー>カテゴリー>検索キーワード、ひらがな読み対応)で絞り込む。フィルター・検索は**すべて掛け合わせる**(AND条件。フィルターの各種類内は複数選択でOR条件)。並び替えを何も設定していなければ、並び順・カテゴリー/サブカテゴリー見出しなど既存の表示は変更しない(検索欄は入力のたびに約280msデバウンスして再取得する`loadItems()`の再取得方式)。標準商品名(`product_master.canonical_name`)ごとに1カードへまとめ、その下に実際の商品名(`items.name`)の区画・在庫ロット(賞味期限が近い順)を表示する。賞味期限は「期限:7/20 (あと3日)」のように日付と残り日数を併記する(`js/items.js` の `formatExpiryLabel`。期限切れは「期限:7/20 (期限切れ)」、未設定は「期限未設定」)。ロットごとに `[-][数量 単位][+]` で数量を増減できる(個数系の単位は1ずつ、定量系(g/ml/kg/L)は100ずつ)。中央の数量表示をタップすると4桁ドラムロールピッカーが開き、直接数量を設定できる。カード見出し(標準商品名)の隣の⚙️ボタンから商品マスタ 詳細/編集シート(画面構成10。商品属性+最低数量。最低数量は標準商品名=カード単位)を開ける。各商品名区画のⓘボタンからは、その商品名だけの購入日・削除を扱う別シート(画面構成10)を開ける(商品マスタ未作成の商品は、従来どおり単一カードで表示され、その1つのボタンに商品属性・最低数量・購入日・削除のすべてがまとまる)
5. **在庫確認の絞り込みオーバーレイ**(`#inventory-filter-overlay`): Amazonアプリの商品検索フィルターに近い2ペイン構成。左側1列にフィルターの種類一覧(現在は「種別」「カテゴリー」「サブカテゴリー」の3つで、この順に階層になっている。今後「登録日」「購入金額」「消費期限」などを追加する場合は`js/items.js`の`inventoryFilterDefs`配列に定義を足すだけでよい構造にしている。ただし現時点で実装しているのはチェックボックス一覧形式の3種類のみ)、右側にその種類の選択肢をチェックボックス形式で表示する(複数選択可、選択済みをもう一度タップすると解除)。種別(`effectiveType()`。商品マスタがあればその`type`、無ければ`items.category`が種別のフォールバック値をそのまま持つ)の選択を変えると対象外になったカテゴリーの選択が、カテゴリーの選択を変えると対象外になったサブカテゴリーの選択が、それぞれ自動で外れる(カスケード)。選択はオーバーレイを開いている間だけの一時状態で、右下の「結果を表示」を押すまでは在庫一覧に反映されない(背景タップ/×で閉じると破棄される)。左下の「フィルターを解除」はその場で選択をクリアするだけでオーバーレイは開いたまま(反映するにはあらためて「結果を表示」を押す)。ボタン群の上には選択中の内容を「[日用品] > [飲料] > [清涼飲料水]」のように案内文で表示する(種別・カテゴリー・サブカテゴリーを" > "で連結、同じ種類内の複数選択は「・」で連結。何も選んでいなければ非表示)。在庫確認画面の「フィルター」ボタンには選択中の合計件数を表示する(例:「フィルター (2)」)
6. **在庫確認の並び替えオーバーレイ**(`#inventory-sort-overlay`): フィルターと同じ2ペイン構成に加え、上部に「すべて」「カテゴリ」「サブカテゴリ」「カード」「カード内」の横並びタブ(`js/items.js`の`INVENTORY_SORT_LEVELS`)があり、この4階層(「カード」=同じサブカテゴリー内でどの商品カードが上に来るか、「カード内」=1枚のカードの中で商品名ごとの区画がどちらが上に来るか)をそれぞれ独立して[期限が近い順/遠い順・登録日が新しい順/古い順・50音順(あ→わ/わ→あ)]から設定できる(`INVENTORY_SORT_CRITERIA`。同じ選択をもう一度タップすると解除)。「すべて」タブ(他のタブと色を変えて区別)だけは実体を持たず、選ぶと4階層すべてに同じ設定を一括反映する特別なタブで、4階層が同じ設定で揃っている場合だけチェック状態や案内文に反映される。何も設定していない階層は、これまで通りの既定の並び順(カテゴリー/サブカテゴリーは文字コード順、カード/カード内は賞味期限が近い順)のまま変わらない。各階層の「代表値」は、その階層の中に含まれる商品・ロットのうち、選んだ並び替えにとって最良のもの(例: 期限が近い順ならもっとも近い期限)を子から親へ伝播させて決める(`js/items.js`の`foldBestInventorySortValue`)。選択は一時状態で、右下の「並び替え」を押すまで反映されない。左下のボタンは「デフォルトに戻す」で、保存済みの既定値(無ければ「並び替えなし」)を選択状態に戻すだけ(パネルは開いたまま)。案内文の隣の「デフォルトに設定」ボタンは、現在選択中の内容(4階層すべて)をブラウザの`localStorage`(`kurasInventorySortDefault`)に保存し、次回以降のアプリ起動時にもこの既定値がそのまま適用される。「登録日」の並び替えは`items.created_at`列(Supabaseの標準的な作成日時列)を利用する(2026-07-20、実データのテーブルに存在することを確認済み)
7. **商品登録オーバーレイ**(`#register-overlay`): AI写真判定・手動登録のどちらもこの共通オーバーレイの中で完結する。商品マスタ 詳細/編集シートと同じく、オーバーレイ背景(シート外側)をタップすると閉じる。
   - **AI写真判定**: 直接カメラ/写真選択を起動し、判定結果が出たらオーバーレイを開いてAI判定結果(`#review-section`)を表示する。各カードには「登録」(その1件だけ登録)「買い物へ」(その1件だけ買い物リストへ追加。在庫ロットは作成・変更せずAIへの再問い合わせもしない)「削除」(AIへの再問い合わせなしで画面から除外)があり、下部の「まとめて登録する」「まとめて買い物リストに追加」はいずれもその時点で画面に残っているカードだけを対象にする。個別・まとめてのいずれかで買い物リストに追加したカードは画面から除外され、以後の「まとめて登録する」の対象にはならない。カードが1件も残らなくなったタイミング(個別処理・まとめて処理・キャンセルのいずれでも)でオーバーレイごと自動的に閉じる
   - **手動登録**: 選ぶと即座にオーバーレイが開き手動フォーム(`#register-manual-section`)を表示する。入力欄の順序は「商品名・単位・数量・消費期限」で、単位・数量は2列の行、消費期限はその下の独立した全幅の行にする(3列に並べるとiPhone 14幅では単位チップ・増減ボタンの文字が折り返してしまうため)。**商品名欄の下には予測変換欄**(`#item-name-suggestions`)があり、入力するたびに「既存の登録済み商品名(`items.name`)」「標準商品名(`product_master.canonical_name`、ひらがな読み`canonical_name_reading`も検索対象)」のうち部分一致するものを最大8件表示する(候補データはオーバーレイを開いた時に一度だけ取得し、以降は入力のたびにクライアント側でフィルタするためAI呼び出し・デバウンスは不要)。予測欄が開いている間は入力欄に`suggestions-open`クラスが付き、入力欄の下端の角丸を消して予測欄と隙間なく接続する(閉じている間は他の入力欄と同じ四隅とも角丸に戻る、Chromeの検索欄と同じ見た目)。並び順は[商品名を優先度上・標準商品名を優先度下]→[各層の中で完全一致>前方一致>部分一致]→[同着は五十音順](商品名欄には基本的に具体的な商品名を入力するため、標準商品名は一致度が高くてもあえて優先度を下げている)。**ただし完全一致(入力欄とまったく同じ文字列)は候補に出さない**(すでに入力欄の内容そのものであり、標準商品名を選んだ直後に予測を表示し続ける仕様と組み合わせると、選んだ本人が候補に残り続けて見えてしまうため)。各候補の先頭には目印として中黒「・」を添える(Google検索の🔍のような機能アイコンではなく、「検索ではなく候補の提示である」ことを示す意味を持たない記号にしている)。標準商品名の候補だけ、テキストの右端に小さく目立たない「標準」バッジを添えて区別する(バッジを左に置くと行ごとに文字の開始位置がずれて読みにくくなるため、あえて右端にしている)。候補をタップすると商品名欄に反映され、通常は予測欄が閉じるが、**標準商品名の候補を選んだ場合だけ予測欄を表示し続ける**(その標準名にちなんだ商品名候補が新たに一致する可能性があるため)。その後に値を変更すると(商品名候補選択後は)再度予測を表示する。**候補(標準商品名・商品名のどちらでも)を選んだ直後だけ**、予測欄に追加で「詳細を入力する」という行を表示する(タップすると商品名欄の末尾に半角カンマ`,`を自動追記し、そのままカーソルを末尾に置く。以後さらに値を変更すると詳細入力欄は消える)。この状態で続けて具体的な内容(例:「小間切れ」)を入力し「豚肉,小間切れ」のようにカンマ区切りで登録すると、**カンマより前だけ**を商品マスタ解決(`resolveProductMaster`)の検索キーに使い(既存の標準商品名・商品名と完全一致すればAIを呼ばない)、実際に保存・表示される商品名(`items.name`)は「豚肉(小間切れ)」のようにカンマの後ろを括弧書きで結合したものになる(`js/items.js`の`parseNameWithDetail`)。カンマを含まない通常の入力では、この変換は行われず従来通り。数量欄は個数系・定量系(g/ml等)を問わず、タップすると4桁ドラムロール(0000〜9999の整数)が下からせり上がって表示され、選択して「完了」で確定する共通ピッカー(`quantityPicker.js`)を使う。数量欄の下には単位チップと同じデザインの`[-][+]`クイック増減ボタンもあり、増減幅は単位が個数系なら1、定量系(g/ml等)なら100(在庫確認画面の`[-][+]`と同じ判定)。ボタンは数量欄と同じ幅で均等2分割・両端揃え(`.qty-quick-adjust-row`)。数量の初期値は単位が個数系なら1、定量系なら100(単位が変わるたびに追従)。ただしピッカーで確定するかクイック増減ボタンを押すと「手動調整済み」扱いになり、以後は単位を変えても自動では上書きしない。単位欄は、商品名から単位候補が1つに確定した場合(例: トマト→「個」)でも、個数・重さ・容量を選び直せるよう常に「個」「g」「ml」チップを表示する(候補が複数ある場合は従来どおりその候補をチップ表示)。**商品名が未入力の段階でも既定で「個」「g」「ml」チップを表示する**(右隣の数量・消費期限欄には常時表示のクイック調整ボタンがあり、単位欄だけ空だと見た目のバランスが崩れるため。この既定表示は`js/units.js`の`updateUnitSuggestions`という手動登録専用のラッパー内で対応しており、単位欄の値そのものは書き換えないため、ユーザーが商品名を入力する前に単位を手入力していてもその値を上書きしない)。価格欄(`#item-price`)と消費・賞味期限欄は2列の行(単位・数量の行と同じ`row2`)にまとめ、価格を左・消費期限を右に配置する。消費・賞味期限欄(ラベルは「消費・賞味期限」。「(任意)」は付けていない)の下には数量と同じデザインの`[-1日][+1日]`クイック調整ボタンがあり、未入力の場合は今日を基準に加減する。

**価格欄**(ラベルは「価格(円・任意)」): 他の数量系入力と違いドラムロールではなく通常の数値キーボード入力(`input type="number"`)にしている(価格は数量と違って決まった刻み幅が無く、大きくばらつく数値になりやすいため)。価格欄の下には、入力した金額の意味を切り替える2択のON/OFFスイッチ風トグル(`#item-price-mode-toggle`。単位チップ・クイック調整ボタンと同じ配色を使い、左右が連結した1つのピルの中で選択中の片側だけ色がつく見た目、数量・期限のボタンのデザインを踏襲)があり、単位が個数系なら[1つ当たり|合計金額]、定量系(g/ml等)なら[100◯当たり(◯は実際の単位)|購入金額]の2択になる(`js/items.js`の`updatePriceModeLabels`。単位欄が変わるたびにラベルも追従する)。**既定(デフォルト)は「合計金額/購入金額」側**(店の値札・レシートは基本的に「今回買った分の金額」を表示することが多く、そのまま入力するだけで済むようにするため)。

実際に`item_lots.price`へ保存する値は、モードに関わらず常に**「1つ当たりの単価」(定量系は100gなど基準量あたりの単価)**に統一する(`js/items.js`の`computeStoredUnitPrice`)。理由: `item_lots.quantity`は消費が進むと減っていく残数のため、合計金額のまま保存すると後から「消費した分の金額」を正しく逆算できなくなる(購入時点の単価を固定しておけば、`消費した数量 × 単価`でいつでも計算できる)。

- 「1つ当たり/100◯当たり」モード: 入力値をそのまま保存する(数量は使わない)
- 「合計金額/購入金額」モード: 個数系は`入力値 ÷ 数量`、定量系は`入力値 × 100 ÷ 数量`を保存する(整数に丸める)

商品名欄の「,」より前(検索キー、`parseNameWithDetail`参照)が既存の商品名または標準商品名と完全一致すると、その平均単価(整数に丸める。商品名が一致すればその商品名だけの平均、無ければ標準商品名が同じすべての商品の平均)を価格欄へ自動入力し、モードも自動で「1つ当たり/100◯当たり」に切り替える(平均値は常に単価のため。一致しなければ価格欄を空にしモードも既定の「合計金額/購入金額」に戻す。`js/items.js`の`loadPriceAverages`/`findPriceSuggestion`/`applyPricePrefill`。オーバーレイを開いた時に一度だけ全ロットの単価を取得しクライアント側で集計するため、AI呼び出し・追加のデバウンスは不要)。価格欄を一度でも手動編集する、またはモードのトグルを一度でもタップすると、以後は自動入力で上書きしない。「登録する」の下には「買い物リストに追加」ボタンもあり、商品名だけを買い物リストへ追加する(在庫ロットは作成・変更しない。単位・数量・期限・価格は使わない)。登録完了後は約700ms後にオーバーレイが自動的に閉じる
   - **買い物リスト「購入済み」からの登録**(`js/shoppingPurchase.js`): 買い物リストの「購入済み」ボタンを押すと、専用のシートではなくこの手動登録と全く同じウィンドウ・入力欄を開く。商品名が既存の`items`と一致する(在庫連動の項目、またはitem_idを持たない項目でも同名の商品が在庫にある)場合は、その商品名・単位(既存の値をそのまま使い、自動推定で上書きしない)・買い物リストで必要だった数量を初期値にする。一致しない場合は商品名・買い物リストの数量だけを初期値にし、単位は手動登録の新規商品と同じ自動提案が働く。消費・賞味期限は常に未入力から始まる。**このモードのときだけ「買い物リストに追加」ボタンを隠す**(買い物リストの項目を消化する場面のため、そこへ追加し直す意味が無い)。商品名の予測変換は通常どおり機能する。登録が完了すると、通常の登録メッセージに加えて対象の買い物リスト行を削除し買い物リストを更新する(種別の自動判定に失敗した場合は、通常の登録と同様に種別を選び直すまで買い物リストの行は削除されない)。キャンセル・背景タップで閉じた場合は買い物リストの行はそのまま残り、次にオーバーレイを開いたときには「買い物リストに追加」ボタンも必ず元通り表示される
8. **消費確認オーバーレイ**(`#consume-overlay`): AI写真判定・手動検索のどちらもこの共通オーバーレイの中で完結する(検索結果が多くなり得るため、専用ページとして分離)。商品マスタ 詳細/編集シートと同じく、オーバーレイ背景(シート外側)をタップすると閉じる。先頭に検索欄(`#consume-search`)があり、入力するたびに約280msデバウンスして検索する(1文字ごとに問い合わせず、かつ入力途中の古いレスポンスは無視するstale-response対策込み)。検索対象・優先度は[商品名]>[標準商品名]>[サブカテゴリー]>[カテゴリー]>[検索キーワード]で、いずれも部分一致(標準商品名・サブカテゴリー・検索キーワードは`product_master`のひらがな読み列、カテゴリーは固定のひらがな対応表でも一致する。商品名自体にひらがな読みは無い)。一致した項目のうち最も優先度が高いものを基準に検索結果を並べる。
   - **手動検索**: 検索結果は在庫確認画面・AI消費レビューと同じ`item-card`の見た目に統一されたカードで、ヘッダー(商品マスタの`icon`または無ければカテゴリー既定の絵文字アイコン・商品名・在庫・もっとも賞味期限が近いロットの賞味期限・開閉シェブロン)をタップすると、同じカードの中でロット調整欄+「確定」ボタンが`fadeSlideIn`アニメーション付きで展開される(他画面の開閉パネルと同じ動き)。展開中のカードは枠線・影を強調しシェブロンが回転するため、どのカードを操作中か一目でわかる。他のカードをタップすると今開いているカードは閉じて新しいカードが展開され、同じカードをもう一度タップするとトグルで閉じる(同時に展開できるのは1件のみ)。この「確定」ボタンは展開中のカード1件だけを対象に消費を確定する(下部の共通確定ボタンとは別物)。確定すると検索語はそのままに検索結果を再取得し、在庫表示・展開状態を作り直す
   - **AI写真判定**: 判定結果に一致した商品は、検索結果のカードとは別に画面下部の共通レビューリスト(`#consume-review-list`)へまとめて追加され(重複追加はしない)、下部の共通「確定」ボタンでリストに残っている全カードをまとめて確定する
   - 両方共通のロットカードの見た目・操作: 在庫確認画面と同じ見た目(手動検索の展開カードのみ商品名ヘッダーは省略。検索結果の行に既に表示されているため)で、賞味期限が異なる同名商品はロットごとに別行になる。**行ごとの数量欄は「消費後にそのロットに残る数量」を表す**(消費する量そのものではない)。初期値はそのロットの在庫数量そのもの(=何もしなければ何も消費しない)。`[-][残り数量 単位][+]`で増減でき(個数系の単位は1ずつ、定量系(g/ml等)は100ずつ、残り数量は0〜元の在庫数量の範囲でクランプ)、数量表示をタップすると4桁ドラムロールピッカーが開き直接入力できる。数量の下の「1/4」「半分」「すべて」ボタンは、押すと残り数量をそのロットの在庫数量に対する割合(1/4・1/2・0)にする(「すべて」=残り0=全部消費)。確定すると、対象カードの各ロットについて残り数量が元の在庫数量から変わっているものだけをその値に更新する(0になったロットは削除、変えていないロットはそのまま)
9. **日付選択モーダル**(`#calendar-overlay`): 自作カレンダー。`.date-display` クラスを持つ input から共通で呼ばれる
10. **商品マスタ 詳細/編集シート**(`#product-detail-overlay`、`js/productDetail.js`): 在庫確認画面の⚙️/ⓘボタンから、対象に応じて3つのモード(`mode`パラメータ)で同じオーバーレイを開く。セクションの表示/非表示・シート見出しはモードごとに切り替わる(`applySectionVisibility()`)
    - **masterモード**(カード見出しの⚙️。標準商品名=カード単位): 見出し「商品属性の詳細」。商品属性(標準商品名・種別・カテゴリー・サブカテゴリー・保管場所・用途・検索キーワード)と、最低数量(`product_master.low_stock_threshold`)を閲覧・編集できる。購入日(在庫ロット)・削除は表示しない(商品名側のⓘに委ねる)。商品マスタが無いと開けないモードのため「商品属性を作成」の空状態は無い
    - **itemモード**(各商品名区画のⓘ。商品マスタがある商品のみ): 見出し「購入日・削除」。その商品名(`items.id`)の在庫ロット一覧(数量・賞味期限・購入日、読み取り専用)と、この商品名(在庫の全ロットを含む)を削除するボタンのみを表示する。商品属性・最低数量は表示しない(カード側の⚙️に委ねる)
    - **fallbackモード**(商品マスタが無い単独カードの1つだけのボタン): 見出し「商品の詳細」。従来通りすべてをまとめて表示する: 最低数量(`items.low_stock_threshold`)・商品属性(未作成なら「商品属性を作成」ボタン)・在庫ロット一覧(購入日つき)・削除ボタン。ここで「商品属性を作成」すると、それまで`items.low_stock_threshold`に設定していた値(0より大きい場合のみ)を新しく作った`product_master.low_stock_threshold`へそのまま引き継ぐ(既存の商品マスタを再利用した場合は、他の商品名と共有中の値を上書きしないよう引き継がない)

    共通: 最低数量は在庫確認画面のロット数量と同じ操作感で、`[-][最低数量 単位][+]`で増減でき(個数系の単位は1ずつ、定量系(g/ml/kg/L)は100ずつ)、中央の数量表示をタップすると4桁ドラムロールピッカーが開き直接設定できる(masterモードの単位は、カード内に定量系の商品名が1つでもあればその単位を代表として使う。無ければ個数系の単位を使う、`js/quantity.js`の`representativeUnitForEntries()`)。AI初期値か手動編集かを🤖/👤バッジで項目単位に区別(商品属性のみ)。標準商品名(表示名)は編集画面で漢字入力欄・ひらがな読み入力欄の2列に分かれており、両方編集できる(ひらがな読みは必須・ひらがな以外の文字はエラー)。詳細表示では標準商品名をタップするとひらがな読みが表示される(トグル)。標準商品名を編集しても、AIが将来同じ標準商品名と判定した際に既存マスタを再利用するための照合キー(`canonical_normalized_name`)は変更されないため、以後もこの編集済みマスタへ正しく合流する。検索キーワードの追加はスマホでの片手操作を考慮し、Enterキーではなく入力欄横の「追加」ボタンで行う
11. **数量ピッカー**(`#quantity-picker-overlay`): 4桁ドラムロール(0000〜9999の整数)で数量を選択する共通オーバーレイ(`quantityPicker.js`)。手動登録の数量欄・在庫確認のロット数量・消費確認カードの消費数量・商品詳細シートの最低数量から共通で呼び出す。呼び出し元は `openQuantityPicker({ initialValue, unit, title, onConfirm })` を呼ぶだけでよい
12. **購入・消費履歴ページ**(`#view-history`、`js/history.js`): オーバーレイではなく在庫確認画面と同じ独立した`.view`(下部ナビには無いため、左上の丸い戻るボタンでホームへ戻す)。ホーム画面の「購入履歴」カードから開く。日付・種別(食品/日用品)・標準商品名・商品名・区分(購入/消費バッジ)・数量(単位付き)・価格・操作の8列を持つHTML`<table>`で一覧表示する(390px幅には収まらないため`.history-table-wrap`で横スクロールさせる、Excelの表のような一覧)。新しい日付順(`occurred_at`降順)、直近300件まで(`HISTORY_LIMIT`)。タイトル行右端の鉛筆ボタン(`#history-edit-toggle-btn`)で編集モードに切り替えると、通常は非表示の各行の削除ボタン(`.del-btn`)が現れ、各セル(日付・種別・標準商品名・商品名・区分・数量・価格)がその場で手入力できる入力欄/セレクトに変わる(`historyEditRowHtml`。値を変更すると`change`イベントで即座に`item_history`を更新し、表示用キャッシュ`lastHistoryData`にも反映する)。削除は`confirm()`で確認後、その1件だけ`item_history`から削除する(誤って記録された履歴の手動修正用)。表(横スクロール領域)の上には「フィルター」「サマリー」「収支表」ボタン(いずれも在庫確認画面と同じ`.inventory-filter-btn`)を並べて設置する。タイトル行の鉛筆ボタンの隣には「拡大表示」ボタン(`#history-expand-btn`)があり、押すとフルスクリーンの`#history-expand-overlay`を開き、その時点の表示内容(フィルター・並び替え適用済み、常に読み取り専用)を大きく表示する。iPhoneのSafariは画面回転ロックAPI(`screen.orientation.lock`)に対応していないため、端末を回転させなくても横長で見られるよう、CSSで表全体を90度回転させて表示する(`#history-expand-rotator`。端末の縦幅=100vhを回転後の横幅として使う)。ただし端末の自動回転が有効な場合、実際に端末を横向きにするとブラウザのビューポート自体が横長になり、この回転を掛けると二重に回転してしまうため、`@media (orientation: landscape)`でビューポートが既に横長の時は回転を無効化し、そのまま画面いっぱいに表示する(自動回転が無効/ロック中の端末では、ユーザーが手で端末を横向きにすることで正しい向きに見える)。ノッチ・ステータスバー・画面角の丸み(セーフエリア)にタイトル文字や閉じるボタンが隠れないよう、`env(safe-area-inset-*)`(`viewport-fit=cover`指定済みのため実機で実際の値を返す)を余白に加算している。**時計回りに90度回転しているため、回転前のleft/top/right/bottomは実際の画面のtop/right/bottom/leftにそれぞれ1つずつずれて対応する**(回転前の「左」に余白を足すと画面の「上端」=ノッチ・ステータスバー(iPhone14で47pt)から離れる、回転前の「右」に余白を足すと画面の「下端」=ホームインジケーター側から離れる、という対応関係。この対応を誤ると回転していない向きの余白を足すことになり隠れたままになるため注意)。env()が実機で正しく取れない場合に備え、`max()`で固定値(画面上端側は55px、他は16〜28px)を下限にしている。横向きのビューポート(実際に端末側で自動回転した場合)ではこの回転が掛からないため、`@media (orientation: landscape)`内では通常通りtop/right/bottom/leftをそのまま対応させている。タイトルは表示せず表の表示領域を最大化しており、閉じるボタン(`.history-expand-close-btn`)はヘッダー行を持たず表の上に絶対配置で浮かせている(位置は`--expand-pad-top`/`--expand-pad-right`というCSS変数を余白設定とボタン位置の両方で共有し、表のセーフエリア余白と常に一致させる)。表のヘッダー行(sticky)と重なっても埋もれないよう、閉じるボタンは`--primary`色の塗りつぶし+白アイコンにして視認性を確保している。閉じるボタン(✕)でオーバーレイを閉じるだけで、通常表の編集モード・フィルター・並び替えの状態には影響しない。
    - **フィルター**(`#history-filter-overlay`): 在庫確認画面の絞り込み(画面構成5)と同じAmazon風2ペイン構成だが、日付・価格はチェックボックスではなく範囲(from/to)入力にする点が異なる。左側の種類一覧は「日付」(開始日・終了日、`.date-display`から自作カレンダーで選択)・「種別」(食品/日用品、チェックボックス)・「標準商品名」(履歴に登場するものすべてをあ→わ順で列挙、チェックボックス・複数選択可)・「区分」(購入/消費、チェックボックス)・「価格」(最小円・最大円)の5種類(`historyFilterDefs`)。種別/標準商品名/区分どうしのカスケードは無く(在庫確認画面のフィルターとの違い)、すべてAND条件で組み合わさる。右下「結果を表示」で確定、左下「フィルターを解除」はその場でステージをクリアするだけ(オーバーレイは開いたまま)。案内文(`#history-filter-summary`)は選択のたびにリアルタイムで更新される。確定した条件は`js/history.js`がexportする`appliedHistoryFilters`(常に同じオブジェクトを書き換える)と`matchesHistoryFilters(row)`で表現され、購入履歴の表だけでなく購入・消費サマリーページ(画面構成13)にもそのまま適用される
    - **並び替え**(`#history-sort-overlay`): 在庫確認画面の並び替え(2ペイン+階層タブ)のようなUIにはせず、商品登録の「1つ当たり/合計金額」と同じデザインの連結スイッチ(`.price-mode-toggle`、register.cssで定義)を3段並べただけの単純な構成にする。「日付」(新しい順/古い順、常時有効)・「区分」(同じ日付内での並び順。指定なし/消費が上/購入が上)・「金額」(指定なし/高い順/安い順、数量×価格)の3項目を独立して設定でき、日付→区分→金額の優先順位で階層的に並べ替える(区分・金額が両方「指定なし」なら、その日付内は取得時点の順序=`occurred_at`降順のまま。`compareHistoryRows`、Array.sortの安定ソートに依存)。下部は「デフォルトに戻す」(ステージのみ初期値に戻す。日付=新しい順・区分/金額=指定なし)と「並び替え」(確定)の2ボタン。確定した内容は`appliedHistorySort`に保持される。**「ホームに戻る」ボタンを押すと、フィルター(`appliedHistoryFilters`)・並び替え(`appliedHistorySort`)はどちらも初期状態にリセットされる**(次に購入履歴を開いた時は必ずフィルター無し・日付新しい順から始まる。`resetHistoryFiltersAndSort`)
    - **サマリー**(`#summary-open-btn`): 押すと購入・消費サマリーページ(画面構成13)を開く
    - **収支表**(`#balance-open-btn`): 押すと在庫収支表ページ(画面構成14)を開く
13. **購入・消費サマリーページ**(`#view-summary`、`js/summary.js`): 購入履歴ページと同じく独立した`.view`(下部ナビには無いため、戻るボタンで購入履歴ページへ戻す)。日付選択用の自作カレンダー(`js/calendar.js`、画面構成9)とは別モジュールの、集計専用の月間カレンダー。前月/次月ボタンで表示月を切り替え、その月の`item_history`を取得して日付ごとに`quantity×price`を合計する(価格が無い記録は金額を計算できないため集計から除く)。カレンダーのマスは縦長(`.summary-cal-day`)にし、日付の数字を上に詰め、その下に該当する日があれば🛒(緑・購入合計)、➖(オレンジ・消費合計)の2行を表示する(どちらも0円の場合はその行自体を表示しない)。月ラベル(`#summary-cal-month-label`)をタップすると、消費期限入力の自作カレンダー(`#calendar-overlay`、`js/calendar.js`)と同じく、画面の上の層に背景をぼかして表示するオーバーレイ(`#summary-yearmonth-overlay`、`z-index:1100`)で年・月をプルダウン選択できる(ページ内に直置きしない)。年の選択肢は実行時の現在年から+10年まで、月は1〜12月。サマリーは常にどこかの月を表示するため「未設定にする」ボタンは無い。年または月のどちらかを選ぶと消費期限カレンダーと同様即座に確定してオーバーレイが閉じる(背景タップ・下部の「閉じる」ボタンでも閉じる。消費期限カレンダーの`.calendar-footer`をそのまま流用し、「未設定にする」ボタンだけ無い)。今日のマスは`--primary`色の枠で強調する。金額(合計)の計算は`js/history.js`の`computeHistoryRowAmount`を共通で使う(定量系(g/ml/kg/L)は`price`が「100◯当たりの単価」のため`数量÷100×単価`、個数系は`数量×単価`。単純な`数量×単価`だと定量系で100倍の金額になってしまうため、この関数を経由せず独自に計算しないこと)。購入履歴ページの「フィルター」で確定した条件(`js/history.js`の`matchesHistoryFilters`)は、月ごとの取得結果にもそのまま適用してから集計する。**日付のマスをタップすると、その日の購入履歴を見られる**(`js/history.js`の`openHistoryForDate`。他に設定中の絞り込み条件はそのまま維持し、日付の範囲だけタップした1日に絞り込んで購入履歴ページへ遷移する)。開始日=終了日が同じ1日に絞り込まれている間だけ、購入履歴ページの表は常に同じ値になり冗長な日付列を隠す(`isSingleDayHistoryView`、`.history-table.day-view`。この条件はサマリー経由に限らず、フィルターの日付欄を手動で同じ日にそろえた場合にも同様に働く)。並び順そのものは単一日かどうかによらず、購入履歴ページの「並び替え」(画面構成12)で常時共通の設定に従う
14. **在庫収支表ページ**(`#view-balance`、`js/balanceSheet.js`): 購入履歴ページの「収支表」ボタンから開く独立した`.view`(下部ナビには無いため、戻るボタンで購入履歴ページへ戻す)。サマリーページと同じ前月/次月ボタンで表示月を切り替え、月ラベル(`#balance-cal-month-label`)のタップでも同じ操作感の年月プルダウン選択(`#balance-yearmonth-overlay`、背景をぼかしたオーバーレイ、年は実行時の現在年から+10年まで)で切り替えられる。貸借対照表のような2カラムの表で、見出し(`借方(在庫)`/`貸方`)は各半分の中央寄せ、左側「借方(在庫)」に食品/日用品それぞれの「今月末在庫」(自動計算)、右側「貸方」に食品/日用品それぞれの「前月繰越」「今月購入」「今月消費」を表示する。**前月繰越**は表示月の開始日より前の`item_history`全件を、購入は+・消費は-として食品/日用品ごとに累積した金額(=その月の期首在庫評価額。集計期間はサマリー・購入履歴のフィルターとは独立で、常に月全体)。**今月末在庫**(借方) = 前月繰越+今月購入-今月消費(`js/balanceSheet.js`の`loadBalanceSheet`/`renderBalanceTable`)。この計算式により、借方の総合計(食品+日用品の今月末在庫)と貸方の総合計(食品+日用品それぞれの前月繰越+今月購入-今月消費)は必ず一致する。今月消費は`▲9,000円`のように会計風のマイナス表記にする(`formatAccountingYen`)。金額計算は`js/history.js`の`computeHistoryRowAmount`を共通で使う(サマリーと同じく定量系の単価換算に対応)

**注意**: モーダル/シート系のオーバーレイは `z-index: 1000`(`#inventory-filter-overlay`・`#register-overlay`・`#consume-overlay`・`#product-detail-overlay` など)を使うため、`#calendar-overlay`・`#quantity-picker-overlay` は、他のシートの中から開かれても常に手前に表示されるよう `z-index: 1100` にしている。新しいオーバーレイを追加する際、他のシートの中から開く可能性があるならこの点に注意する。
