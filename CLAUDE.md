# CLAUDE.md

このファイルは、このリポジトリで作業する Claude Code(および開発者)向けのガイドです。

## プロジェクト概要

**KURA:S** ― 家庭内の在庫(食材・日用品)を管理するモバイル向けWebアプリ(PWA的なSPA)。
AI写真判定・手動入力で在庫を登録/消費でき、在庫が少なくなると買い物リストに自動連携する。

- 認証・DB: Supabase
- AI商品判定: Google Gemini API(`gemini-3.1-flash-lite`)
- ビルドツールなし。素のHTML/CSS/JS(ESモジュール)のみで構成された静的サイト

## 開発ルール

### 基本方針

- **保守性・拡張性を最優先**する。奇抜な最適化やトリッキーな書き方より、誰が読んでも分かるコードを優先する
- **既存機能・既存UIを壊さない**。動作確認が難しい変更は、影響範囲を明示したうえで慎重に進める
- **必要最小限のファイルのみ編集**する。関係のないファイルには触れない
- **変更範囲を最小限にする**。ついでの修正・ついでの整形はしない
- **リファクタリング時は見た目・動作を一切変更しない**。見た目や挙動が変わる場合はリファクタリングと呼ばない
- **実装前に必ず設計案を提示**し、私の承認後に実装する
- **大規模な変更は小さなステップに分けて**進める。1つのステップごとに動作確認できる単位にする
- **実装後は必ずローカルで動作確認**し、結果を報告する

### 編集プロセス(必ず守ること)

1. **実装前に、変更方針・変更対象ファイル・実行予定コマンドを含む設計案を提示する**
2. **私(ユーザー)の承認を得る**
3. **承認後は、設計案の範囲内であれば、個々の編集・コマンド実行のたびに確認を求めず、まとめて進める**
4. **不要なリネームを行わない**(ファイル名・変数名・関数名など)
5. **不要なフォーマット変更を行わない**(インデント統一、改行整形、クォート統一など、指示されていない整形はしない)

設計案の範囲を超える変更が必要になった場合(対象外ファイルの編集、方針と異なるアプローチへの変更など)は、あらためて確認を取る。

### AI実装

- AI呼び出し回数を最小限にする
- キャッシュを最優先で利用する
- 商品マスタを優先利用する
- 将来の拡張性を考慮した設計にする
- 商品マスタ未ヒット時にAIへ標準商品名を判定させる際は、既存の標準商品名・検索キーワードとの文字列類似度(bigram重なり係数、`js/productMaster.js` の `fetchKnownCanonicalNameCandidates`)でローカルに絞り込んだ候補をヒントとして渡し、表記ゆれ(例:「豚小間切れ」⇔「豚小間切れ肉」)で同じ標準商品名を再利用しやすくする。この絞り込みはAIを呼ばずに行うため、AI呼び出し回数は増えない
- 使用モデルは `gemini-3.1-flash-lite` に固定している(2026-07-16)。無料枠の1日あたりリクエスト上限が少ない `gemini-flash-latest`(Gemini 3.5 Flash系、無料枠20回/日)で上限超過エラーが頻発したため切替。別モデルへの自動フォールバックは実装しない方針
- 標準商品名・サブカテゴリー・検索キーワードのひらがな読みは、商品属性生成の同じAI呼び出し(`identifyProductAttributes`)にまとめて含めており、専用の追加呼び出しは行わない。カテゴリーはenumが固定のためAIを使わず`js/productMaster.js`の固定表(`getCategoryReading`)で読みを持つ

- デザインコンセプトは **「PayPayの使いやすさ」× 「ME:Iの未来感・高級感」**

  - ME:I的: 上品なパープル基調、繊細な余白とタイポグラフィ、控えめで上質なアニメーション
- **シンプルで、情報量が多くても見やすいUI**にする(在庫一覧・買い物リストなど情報密度が上がりやすい画面で特に意識する)
- **スマホファースト**(`.wrap` の `max-width: 460px` を前提に設計。PC幅対応は二の次)
- **アクセシビリティ**を意識する(`aria-label`、タップ領域の大きさ、コントラスト、フォーカス可視性など)

## コード方針

### CSS(`css/`)
- **画面・コンポーネント単位で管理**する(例: `home.css` はホーム画面、`modal.css` はモーダル共通部品)
- 共通のトークン(色・角丸・影など)は `common.css` の CSS変数(`:root`)にまとめる。新しい色や数値を各ファイルにベタ書きしない
- 新しい画面・コンポーネントを追加する場合は、既存の分割方針に合わせて新規CSSファイルを作り、`index.html` の `<head>` に `<link>` を追加する

### JavaScript(`js/`)
- **機能単位でモジュールを分割する**(例: `consume.js` = 消費関連、`shopping.js` = 買い物リスト関連)。既存の分割粒度に合わせる
- **共通処理は再利用できるようにまとめる**(`utils.js`, `elements.js` などに集約。同じロジックをコピペしない)
- 新規モジュールは `js/main.js` に副作用importとして追加する(既存の初期化パターンに合わせる)
- モジュール間の依存は `export` / `import` を使う(グローバル変数は最小限に)。ただし `onclick="..."` 属性から呼ぶ必要がある関数は `window.xxx` を使う既存パターンを踏襲する
- **コメントは必要最小限**。「何をしているか」ではなく「なぜそうしているか」が非自明な場合のみ書く
- **読みやすいコード**を優先する(早すぎる抽象化・過剰な汎用化はしない)

## データモデル(在庫まわり)

在庫は3層に分かれている。

- **`items`** = 商品(名前・分類・単位・最低数量・`product_master_id`)。数量・賞味期限は持たない(`quantity`/`expiry_date`列は`item_lots`への移行完了により削除済み)
- **`item_lots`** = 在庫ロット(数量・賞味期限・購入日)。1商品(`items`)に対して複数持てる。**「同じ商品 かつ 同じ賞味期限」のときだけ既存ロットに数量を加算し、賞味期限が異なれば必ず新規ロットになる**。ロットの取得・表示・消費順は常に `expiry_date ASC NULLS LAST`(賞味期限が近い順、未設定は最後)
- **`product_master`** / **`product_name_alias`** = 商品属性(標準商品名・カテゴリー・サブカテゴリー・保存場所・用途・検索キーワードなど、AIが生成)。ユーザー・世帯をまたいで共有されるマスタで、在庫のライフサイクル(登録・消費・削除)とは独立している。標準商品名(`canonical_name`)・サブカテゴリー(`sub_category`)・検索キーワード(`search_keywords`)にはひらがな読み列(`canonical_name_reading`・`sub_category_reading`・`search_keywords_reading`)があり、AI属性生成時に同じ呼び出しでまとめて生成される(カテゴリーのひらがな読みは固定enumのため、AIを使わず`js/productMaster.js`の`getCategoryReading`の固定表で持つ)。標準商品名(`canonical_name`、表示用)は商品マスタ 詳細/編集シートから編集できるが、`canonical_normalized_name`(AIが将来同じ標準商品名と判定した際に既存マスタを再利用するための照合キー)は編集の影響を受けない

在庫確認画面は、`items.product_master_id` が同じ商品を「標準商品名」(`product_master.canonical_name`)の1カードにまとめ、その下に実際の商品名(`items.name`)ごとの区画を表示する(商品マスタが無い商品は、従来どおり単一カードで表示するフォールバックになる)。カテゴリー見出しは `product_master.category`(無ければ`items.category`)、サブカテゴリー見出しは `product_master.sub_category`(無ければ「未分類」)を使う。

数量の増減・削除は必ず `item_lots.id` を主キーに操作する(ロットの数量が0になったらそのロットだけ削除し、他のロット・他の商品には影響させない)。買い物リストとの連動(在庫少なめによる自動追加/自動削除)は `js/shopping.js` の `syncShoppingListForItem()` に一本化されており、在庫の増減・消費・登録などすべての経路がこの関数を経由する。判定式は `low_stock_threshold > 0 && totalQuantity(全ロット合計) < low_stock_threshold`(最低数量が`0`の商品は在庫が`0`でも自動追加しない、という運用上の意図を表す)。

新規商品登録時の `items.category` は、ユーザーが手入力するのではなく、商品マスタ解決(`resolveProductMaster`)が判定した `product_master.type`(食品/日用品)を自動的に反映する(`js/items.js` の `resolveItem`)。商品マスタの判定に失敗した場合のみ `product_master_id` は `null` のままとなり、登録画面に表示される「種別」フォールバック入力欄の値がそのまま `items.category` に入る。

## 画面構成

**共通通知欄**(`#app-notice`): ログイン後の`.app-header`(KURA:Sロゴ)と**同じカードの中**に置かれた通知行で、新しいカードは作らず`.app-header`自体の縦幅が伸びて表示される(`.app-header-top`がロゴ+ログアウト行、その下に`#app-notice`)。登録・消費・買い物リスト追加・商品属性の生成/再生成など、完了時の案内メッセージ(旧: 各パネル内に緑字で表示していたもの)はここに一本化して表示する(小さめの文字・表示後4秒で自動的に消える)。呼び出しは`js/utils.js`の`showAppNotice(text)`。商品登録時は「AIが商品属性を生成しました」/「既存の商品属性を利用しました」を先頭に付け、AIが新規に属性を作ったか既存を再利用したかを区別して表示する(`upsertItemByName`/`resolveItem`が返す`productMasterStatus`で判定)。入力エラーなど失敗時のメッセージは対象外で、従来どおり各パネル内に赤字で表示する。ログイン画面(`#auth-card`)はヘッダーが別(`.brand-header`)のため対象外。

1. **認証画面**(`#auth-card`): ログイン / 新規登録(Supabase Auth)
2. **ホーム**(`#view-home`):
   - 挨拶・日付表示
   - 「商品を登録」「商品を消費」の2枚のアコーディオンカード。開くとAI/手動の2ボタンが並ぶだけで、選ぶとどちらもその場では差し替わらず共通オーバーレイを開く(登録は`#register-overlay`・詳細は画面構成5、消費は`#consume-overlay`・詳細は画面構成6)。中身(AI判定結果・手動フォーム・検索・確定)の詳細はそれぞれの画面構成の項目を参照
   - 登録画面(AI・手動とも)に「分類」の手入力欄は無い。新規商品は商品マスタ(AI)が判定した`type`(食品/日用品)を`items.category`へ自動反映する。判定に失敗した場合のみ、その場に「種別」選択欄(食品/日用品)が表示され、選び直して再度「登録」を押すとAIを再呼び出しせずその値で登録される(個別登録・まとめて登録の両方に対応。まとめて登録では種別入力が必要な商品だけカードが残る)
   - 賞味期限のお知らせカード(現在はダミー表示)
   - 今後追加予定カード(AIレシピ・家族共有・購入履歴)
3. **買い物リスト**(`#view-shopping`): 買い物項目一覧のみで、画面上からの自由入力欄は無い(項目はホーム画面の「商品を登録」の手動/AIから追加するか、在庫少なめによる自動追加で増える)。「購入済み」ボタンを押すと商品登録オーバーレイ(画面構成5の「買い物リスト「購入済み」からの登録」)を開く(即座には確定しない)。各行には「削除」ボタンがある(編集機能は無い)
4. **在庫確認**(`#view-inventory`): カテゴリー/サブカテゴリーでフィルタ可能な商品一覧。標準商品名(`product_master.canonical_name`)ごとに1カードへまとめ、その下に実際の商品名(`items.name`)の区画・在庫ロット(賞味期限が近い順)を表示する。賞味期限は「期限:7/20 (あと3日)」のように日付と残り日数を併記する(`js/items.js` の `formatExpiryLabel`。期限切れは「期限:7/20 (期限切れ)」、未設定は「期限未設定」)。ロットごとに `[-][数量 単位][+]` で数量を増減できる(個数系の単位は1ずつ、定量系(g/ml/kg/L)は100ずつ)。中央の数量表示をタップすると4桁ドラムロールピッカーが開き、直接数量を設定できる(最低数量・商品削除はカードになく「詳細」画面にある)。各商品名区画の「詳細」ボタンから商品マスタ 詳細/編集シートを開ける(商品マスタ未作成の商品は、従来どおり単一カードで表示される)
5. **商品登録オーバーレイ**(`#register-overlay`): AI写真判定・手動登録のどちらもこの共通オーバーレイの中で完結する。商品マスタ 詳細/編集シートと同じく、オーバーレイ背景(シート外側)をタップすると閉じる。
   - **AI写真判定**: 直接カメラ/写真選択を起動し、判定結果が出たらオーバーレイを開いてAI判定結果(`#review-section`)を表示する。各カードには「登録」(その1件だけ登録)「買い物へ」(その1件だけ買い物リストへ追加。在庫ロットは作成・変更せずAIへの再問い合わせもしない)「削除」(AIへの再問い合わせなしで画面から除外)があり、下部の「まとめて登録する」「まとめて買い物リストに追加」はいずれもその時点で画面に残っているカードだけを対象にする。個別・まとめてのいずれかで買い物リストに追加したカードは画面から除外され、以後の「まとめて登録する」の対象にはならない。カードが1件も残らなくなったタイミング(個別処理・まとめて処理・キャンセルのいずれでも)でオーバーレイごと自動的に閉じる
   - **手動登録**: 選ぶと即座にオーバーレイが開き手動フォーム(`#register-manual-section`)を表示する。入力欄の順序は「商品名・単位・数量・消費期限」。**商品名欄の下には予測変換欄**(`#item-name-suggestions`)があり、入力するたびに「既存の登録済み商品名(`items.name`)」「標準商品名(`product_master.canonical_name`、ひらがな読み`canonical_name_reading`も検索対象)」のうち部分一致するものを最大8件表示する(候補データはオーバーレイを開いた時に一度だけ取得し、以降は入力のたびにクライアント側でフィルタするためAI呼び出し・デバウンスは不要)。予測欄が開いている間は入力欄に`suggestions-open`クラスが付き、入力欄の下端の角丸を消して予測欄と隙間なく接続する(閉じている間は他の入力欄と同じ四隅とも角丸に戻る、Chromeの検索欄と同じ見た目)。並び順は[商品名を優先度上・標準商品名を優先度下]→[各層の中で完全一致>前方一致>部分一致]→[同着は五十音順](商品名欄には基本的に具体的な商品名を入力するため、標準商品名は一致度が高くてもあえて優先度を下げている)。**ただし完全一致(入力欄とまったく同じ文字列)は候補に出さない**(すでに入力欄の内容そのものであり、標準商品名を選んだ直後に予測を表示し続ける仕様と組み合わせると、選んだ本人が候補に残り続けて見えてしまうため)。各候補の先頭には目印として中黒「・」を添える(Google検索の🔍のような機能アイコンではなく、「検索ではなく候補の提示である」ことを示す意味を持たない記号にしている)。標準商品名の候補だけ、テキストの右端に小さく目立たない「標準」バッジを添えて区別する(バッジを左に置くと行ごとに文字の開始位置がずれて読みにくくなるため、あえて右端にしている)。候補をタップすると商品名欄に反映され、通常は予測欄が閉じるが、**標準商品名の候補を選んだ場合だけ予測欄を表示し続ける**(その標準名にちなんだ商品名候補が新たに一致する可能性があるため)。その後に値を変更すると(商品名候補選択後は)再度予測を表示する。数量欄は個数系・定量系(g/ml等)を問わず、タップすると4桁ドラムロール(0000〜9999の整数)が下からせり上がって表示され、選択して「完了」で確定する共通ピッカー(`quantityPicker.js`)を使う。数量欄の下には単位チップと同じデザインの`[-][+]`クイック増減ボタンもあり、増減幅は単位が個数系なら1、定量系(g/ml等)なら100(在庫確認画面の`[-][+]`と同じ判定)。ボタンは数量欄と同じ幅で均等2分割・両端揃え(`.qty-quick-adjust-row`)。数量の初期値は単位が個数系なら1、定量系なら100(単位が変わるたびに追従)。ただしピッカーで確定するかクイック増減ボタンを押すと「手動調整済み」扱いになり、以後は単位を変えても自動では上書きしない。単位欄は、商品名から単位候補が1つに確定した場合(例: トマト→「個」)でも、個数・重さ・容量を選び直せるよう常に「個」「g」「ml」チップを表示する(候補が複数ある場合は従来どおりその候補をチップ表示)。**商品名が未入力の段階でも既定で「個」「g」「ml」チップを表示する**(右隣の数量・消費期限欄には常時表示のクイック調整ボタンがあり、単位欄だけ空だと見た目のバランスが崩れるため。この既定表示は`js/units.js`の`updateUnitSuggestions`という手動登録専用のラッパー内で対応しており、単位欄の値そのものは書き換えないため、ユーザーが商品名を入力する前に単位を手入力していてもその値を上書きしない)。消費・賞味期限欄の下にも数量と同じデザインの`[-1日][+1日]`クイック調整ボタンがあり、未入力の場合は今日を基準に加減する。「登録する」の下には「買い物リストに追加」ボタンもあり、商品名だけを買い物リストへ追加する(在庫ロットは作成・変更しない。単位・数量・期限は使わない)。登録完了後は約700ms後にオーバーレイが自動的に閉じる
   - **買い物リスト「購入済み」からの登録**(`js/shoppingPurchase.js`): 買い物リストの「購入済み」ボタンを押すと、専用のシートではなくこの手動登録と全く同じウィンドウ・入力欄を開く。商品名が既存の`items`と一致する(在庫連動の項目、またはitem_idを持たない項目でも同名の商品が在庫にある)場合は、その商品名・単位(既存の値をそのまま使い、自動推定で上書きしない)・買い物リストで必要だった数量を初期値にする。一致しない場合は商品名・買い物リストの数量だけを初期値にし、単位は手動登録の新規商品と同じ自動提案が働く。消費・賞味期限は常に未入力から始まる。**このモードのときだけ「買い物リストに追加」ボタンを隠す**(買い物リストの項目を消化する場面のため、そこへ追加し直す意味が無い)。商品名の予測変換は通常どおり機能する。登録が完了すると、通常の登録メッセージに加えて対象の買い物リスト行を削除し買い物リストを更新する(種別の自動判定に失敗した場合は、通常の登録と同様に種別を選び直すまで買い物リストの行は削除されない)。キャンセル・背景タップで閉じた場合は買い物リストの行はそのまま残り、次にオーバーレイを開いたときには「買い物リストに追加」ボタンも必ず元通り表示される
6. **消費確認オーバーレイ**(`#consume-overlay`): AI写真判定・手動検索のどちらもこの共通オーバーレイの中で完結する(検索結果が多くなり得るため、専用ページとして分離)。商品マスタ 詳細/編集シートと同じく、オーバーレイ背景(シート外側)をタップすると閉じる。先頭に検索欄(`#consume-search`)があり、入力するたびに約280msデバウンスして検索する(1文字ごとに問い合わせず、かつ入力途中の古いレスポンスは無視するstale-response対策込み)。検索対象・優先度は[商品名]>[標準商品名]>[サブカテゴリー]>[カテゴリー]>[検索キーワード]で、いずれも部分一致(標準商品名・サブカテゴリー・検索キーワードは`product_master`のひらがな読み列、カテゴリーは固定のひらがな対応表でも一致する。商品名自体にひらがな読みは無い)。一致した項目のうち最も優先度が高いものを基準に検索結果を並べる。
   - **手動検索**: 検索結果は在庫確認画面・AI消費レビューと同じ`item-card`の見た目に統一されたカードで、ヘッダー(商品マスタの`icon`または無ければカテゴリー既定の絵文字アイコン・商品名・在庫・もっとも賞味期限が近いロットの賞味期限・開閉シェブロン)をタップすると、同じカードの中でロット調整欄+「確定」ボタンが`fadeSlideIn`アニメーション付きで展開される(他画面の開閉パネルと同じ動き)。展開中のカードは枠線・影を強調しシェブロンが回転するため、どのカードを操作中か一目でわかる。他のカードをタップすると今開いているカードは閉じて新しいカードが展開され、同じカードをもう一度タップするとトグルで閉じる(同時に展開できるのは1件のみ)。この「確定」ボタンは展開中のカード1件だけを対象に消費を確定する(下部の共通確定ボタンとは別物)。確定すると検索語はそのままに検索結果を再取得し、在庫表示・展開状態を作り直す
   - **AI写真判定**: 判定結果に一致した商品は、検索結果のカードとは別に画面下部の共通レビューリスト(`#consume-review-list`)へまとめて追加され(重複追加はしない)、下部の共通「確定」ボタンでリストに残っている全カードをまとめて確定する
   - 両方共通のロットカードの見た目・操作: 在庫確認画面と同じ見た目(手動検索の展開カードのみ商品名ヘッダーは省略。検索結果の行に既に表示されているため)で、賞味期限が異なる同名商品はロットごとに別行になる。**行ごとの数量欄は「消費後にそのロットに残る数量」を表す**(消費する量そのものではない)。初期値はそのロットの在庫数量そのもの(=何もしなければ何も消費しない)。`[-][残り数量 単位][+]`で増減でき(個数系の単位は1ずつ、定量系(g/ml等)は100ずつ、残り数量は0〜元の在庫数量の範囲でクランプ)、数量表示をタップすると4桁ドラムロールピッカーが開き直接入力できる。数量の下の「1/4」「半分」「すべて」ボタンは、押すと残り数量をそのロットの在庫数量に対する割合(1/4・1/2・0)にする(「すべて」=残り0=全部消費)。確定すると、対象カードの各ロットについて残り数量が元の在庫数量から変わっているものだけをその値に更新する(0になったロットは削除、変えていないロットはそのまま)
7. **日付選択モーダル**(`#calendar-overlay`): 自作カレンダー。`.date-display` クラスを持つ input から共通で呼ばれる
8. **商品マスタ 詳細/編集シート**(`#product-detail-overlay`): 最低数量(在庫設定)・標準商品名・種別・カテゴリー・サブカテゴリー・保管場所・用途・検索キーワードを閲覧・編集できる。最低数量は在庫確認画面のロット数量と同じ操作感で、`[-][最低数量 単位][+]`で増減でき(個数系の単位は1ずつ、定量系(g/ml/kg/L)は100ずつ)、中央の数量表示をタップすると4桁ドラムロールピッカーが開き直接設定できる。AI初期値か手動編集かを🤖/👤バッジで項目単位に区別。標準商品名(表示名)は編集画面で漢字入力欄・ひらがな読み入力欄の2列に分かれており、両方編集できる(ひらがな読みは必須・ひらがな以外の文字はエラー)。詳細表示では標準商品名をタップするとひらがな読みが表示される(トグル)。標準商品名を編集しても、AIが将来同じ標準商品名と判定した際に既存マスタを再利用するための照合キー(`canonical_normalized_name`)は変更されないため、以後もこの編集済みマスタへ正しく合流する。検索キーワードの追加はスマホでの片手操作を考慮し、Enterキーではなく入力欄横の「追加」ボタンで行う。商品属性が未作成の商品には「商品属性を作成」ボタンを表示する。商品(在庫の全ロットを含む)を削除するボタンもここにある
9. **数量ピッカー**(`#quantity-picker-overlay`): 4桁ドラムロール(0000〜9999の整数)で数量を選択する共通オーバーレイ(`quantityPicker.js`)。手動登録の数量欄・在庫確認のロット数量・消費確認カードの消費数量・商品詳細シートの最低数量から共通で呼び出す。呼び出し元は `openQuantityPicker({ initialValue, unit, title, onConfirm })` を呼ぶだけでよい

**注意**: モーダル/シート系のオーバーレイは `z-index: 1000`(`#register-overlay`・`#consume-overlay`・`#product-detail-overlay` など)を使うため、`#calendar-overlay`・`#quantity-picker-overlay` は、他のシートの中から開かれても常に手前に表示されるよう `z-index: 1100` にしている。新しいオーバーレイを追加する際、他のシートの中から開く可能性があるならこの点に注意する。
