# AGENTS.md

このファイルは、このリポジトリで作業する Codex(および開発者)向けのガイドです。

## プロジェクト概要

**KURA:S** ― 家庭内の在庫(食材・日用品)を管理するモバイル向けWebアプリ(PWA的なSPA)。
AI写真判定・手動入力で在庫を登録/消費でき、在庫が少なくなると買い物リストに自動連携する。

- 認証・DB: Supabase
- AI商品判定: Google Gemini API(`gemini-3.1-flash-lite`)
- ビルドツールなし。素のHTML/CSS/JS(ESモジュール)のみで構成された静的サイト

## 開発ルール

### 基本方針

- **保守性・拡張性を最優先**する。奇抜な最適化やトリッキーな書き方より、誰が読んでも分かるコードを優先する
- **既存機能・既存UIを壊さない**。動作確認が難しい変更は、影響範囲を明示したうえで慎重に進める
- **必要最小限のファイルのみ編集**する。関係のないファイルには触れない
- **変更範囲を最小限にする**。ついでの修正・ついでの整形はしない
- **リファクタリング時は見た目・動作を一切変更しない**。見た目や挙動が変わる場合はリファクタリングと呼ばない
- **実装前に必ず設計案を提示**し、私の承認後に実装する
- **大規模な変更は小さなステップに分けて**進める。1つのステップごとに動作確認できる単位にする
- **実装後は必ずローカルで動作確認**し、結果を報告する

### 編集プロセス(必ず守ること)

1. **編集前に、変更対象ファイルを一覧表示する**
2. **編集前に、実行予定のコマンドを表示する**
3. **私(ユーザー)の許可を得てから、実際の編集・コマンド実行を行う**
4. **不要なリネームを行わない**(ファイル名・変数名・関数名など)
5. **不要なフォーマット変更を行わない**(インデント統一、改行整形、クォート統一など、指示されていない整形はしない)

上記は、明示的に「まとめて進めてOK」「確認不要」などの指示がない限り、常に適用する。

### データベース

- データベース構造を変更する場合は、必ず `sql` フォルダにSQLファイルを作成する
- SQLは連番付きで管理する

```
sql/

  002_create_product_name_alias.sql
  003_add_product_master_id_to_items.sql
  004_add_product_master_rls.sql
  005_add_product_master_update_policies.sql
  006_add_product_master_icon_and_edited_fields.sql
  007_create_item_lots.sql
  008_drop_barcode_column_from_items.sql
  009_drop_tag_column_from_items.sql
```

- SQLだけをチャットへ貼り付けるのではなく、必ずSQLファイルも作成する
- Supabaseで実行が必要な場合は、「Supabaseでこのsqlを実行してください」と案内する

### Git

- 機能単位でコミットできるように実装する
- 作業終了時に推奨コミットメッセージを提案する

### AI実装

- AI呼び出し回数を最小限にする
- キャッシュを最優先で利用する
- 商品マスタを優先利用する
- 将来の拡張性を考慮した設計にする
- 使用モデルは `gemini-3.1-flash-lite` に固定している(2026-07-16)。無料枠の1日あたりリクエスト上限が少ない `gemini-flash-latest`(Gemini 3.5 Flash系、無料枠20回/日)で上限超過エラーが頻発したため切替。別モデルへの自動フォールバックは実装しない方針

### 作業終了時

毎回以下を実施すること

- AGENTS.mdを最新状態へ更新する
- 変更内容を要約する(何を、なぜ変更したか)
- 変更したファイル一覧を表示する
- 動作確認結果を報告する
- 改善案があれば提案するが、指示なく勝手に実装しない
- 次回おすすめの作業を3件提案する

## UI/UX方針

- デザインコンセプトは **「PayPayの使いやすさ」× 「ME:Iの未来感・高級感」**
  - PayPay的: タップ数が少ない、迷わない、目的の操作に最短で辿り着ける導線
  - ME:I的: 上品なパープル基調、繊細な余白とタイポグラフィ、控えめで上質なアニメーション
- **シンプルで、情報量が多くても見やすいUI**にする(在庫一覧・買い物リストなど情報密度が上がりやすい画面で特に意識する)
- **スマホファースト**(`.wrap` の `max-width: 460px` を前提に設計。PC幅対応は二の次)
- **アクセシビリティ**を意識する(`aria-label`、タップ領域の大きさ、コントラスト、フォーカス可視性など)

## コード方針

### CSS(`css/`)
- **画面・コンポーネント単位で管理**する(例: `home.css` はホーム画面、`modal.css` はモーダル共通部品)
- 共通のトークン(色・角丸・影など)は `common.css` の CSS変数(`:root`)にまとめる。新しい色や数値を各ファイルにベタ書きしない
- 新しい画面・コンポーネントを追加する場合は、既存の分割方針に合わせて新規CSSファイルを作り、`index.html` の `<head>` に `<link>` を追加する

### JavaScript(`js/`)
- **機能単位でモジュールを分割する**(例: `consume.js` = 消費関連、`shopping.js` = 買い物リスト関連)。既存の分割粒度に合わせる
- **共通処理は再利用できるようにまとめる**(`utils.js`, `elements.js` などに集約。同じロジックをコピペしない)
- 新規モジュールは `js/main.js` に副作用importとして追加する(既存の初期化パターンに合わせる)
- モジュール間の依存は `export` / `import` を使う(グローバル変数は最小限に)。ただし `onclick="..."` 属性から呼ぶ必要がある関数は `window.xxx` を使う既存パターンを踏襲する
- **コメントは必要最小限**。「何をしているか」ではなく「なぜそうしているか」が非自明な場合のみ書く
- **読みやすいコード**を優先する(早すぎる抽象化・過剰な汎用化はしない)

## セキュリティ上の既知事項(取り扱い注意)

- `js/config.js` に Supabase anon key と **Gemini APIキーが直書き**されている。このリポジトリは Public のため、新しいキーや秘匿情報を追加する際は特に注意する
- `items` / `shopping_list` テーブルのユーザー間分離(RLS)状況は未確認。マルチユーザー対応に関わる変更をする際は、この点をユーザーに確認してから進める
- バーコードスキャン機能(`html5-qrcode`・OpenFoodFacts照会・`items.barcode`列)は廃止済み(2026-07-16)。関連コード・DB列は削除済み
- タグ機能(`items.tag`・`tags.js`・タグ別フィルター)は廃止済み(2026-07-16)。カテゴリー(`product_master.category`、無ければ`items.category`)・サブカテゴリー(`product_master.sub_category`)による分類へ統一済み

## データモデル(在庫まわり)

在庫は3層に分かれている。

- **`items`** = 商品(名前・分類・単位・最低数量・`product_master_id`)。数量・賞味期限は持たない
- **`item_lots`** = 在庫ロット(数量・賞味期限・購入日)。1商品(`items`)に対して複数持てる。**「同じ商品 かつ 同じ賞味期限」のときだけ既存ロットに数量を加算し、賞味期限が異なれば必ず新規ロットになる**。ロットの取得・表示・消費順は常に `expiry_date ASC NULLS LAST`(賞味期限が近い順、未設定は最後)
- **`product_master`** / **`product_name_alias`** = 商品属性(標準商品名・カテゴリー・サブカテゴリー・保存場所・用途・検索キーワードなど、AIが生成)。ユーザー・世帯をまたいで共有されるマスタで、在庫のライフサイクル(登録・消費・削除)とは独立している

在庫確認画面は、`items.product_master_id` が同じ商品を「標準商品名」(`product_master.canonical_name`)の1カードにまとめ、その下に実際の商品名(`items.name`)ごとの区画を表示する(商品マスタが無い商品は、従来どおり単一カードで表示するフォールバックになる)。カテゴリー見出しは `product_master.category`(無ければ`items.category`)、サブカテゴリー見出しは `product_master.sub_category`(無ければ「未分類」)を使う。

数量の増減・削除は必ず `item_lots.id` を主キーに操作する(ロットの数量が0になったらそのロットだけ削除し、他のロット・他の商品には影響させない)。買い物リストとの連動(在庫少なめ判定)は、商品の全ロット合計と `items.low_stock_threshold` を比較して行う。

`items.quantity` / `items.expiry_date` 列は段階的移行のためまだ残っているが、コードからは参照していない(将来 `item_lots` への移行が十分定着した段階で削除を検討する)。

## 画面構成

1. **認証画面**(`#auth-card`): ログイン / 新規登録(Supabase Auth)
2. **ホーム**(`#view-home`):
   - 挨拶・日付表示
   - 「商品を登録」「商品を消費」の2枚のアコーディオンカード。開くとAI/手動の2ボタンが並び、選ぶとその場で内容が差し替わる
     - AI: ウィンドウを表示せず直接カメラ/写真選択を起動し、判定結果が出たら2ボタンをその場で差し替える。判定結果の各カードには「登録」(その1件だけ登録)と「削除」(AIへの再問い合わせなしで画面から除外)があり、下部の「まとめて登録する」はその時点で画面に残っているカードだけを対象にする
     - 手動: 2ボタンの位置に手動フォーム/検索欄をその場で表示する
   - 賞味期限のお知らせカード(現在はダミー表示)
   - 今後追加予定カード(AIレシピ・家族共有・購入履歴)
3. **買い物リスト**(`#view-shopping`): 自由入力 + 在庫連動の買い物項目一覧
4. **在庫確認**(`#view-inventory`): カテゴリー/サブカテゴリーでフィルタ可能な商品一覧。標準商品名(`product_master.canonical_name`)ごとに1カードへまとめ、その下に実際の商品名(`items.name`)の区画・在庫ロット(賞味期限が近い順)を表示する。ロットごとに `[-][数量 単位][+]` で数量を増減できる(最低数量・商品削除はカードになく「詳細」画面にある)。各商品名区画の「詳細」ボタンから商品マスタ 詳細/編集シートを開ける(商品マスタ未作成の商品は、従来どおり単一カードで表示される)
5. **日付選択モーダル**(`#calendar-overlay`): 自作カレンダー。`.date-display` クラスを持つ input から共通で呼ばれる
6. **商品マスタ 詳細/編集シート**(`#product-detail-overlay`): 最低数量(在庫設定)・標準商品名・種別・カテゴリー・サブカテゴリー・保管場所・用途・検索キーワードを閲覧・編集できる。AI初期値か手動編集かを🤖/👤バッジで項目単位に区別。検索キーワードの追加はスマホでの片手操作を考慮し、Enterキーではなく入力欄横の「追加」ボタンで行う。商品属性が未作成の商品には「商品属性を作成」ボタンを表示する。商品(在庫の全ロットを含む)を削除するボタンもここにある

下部ナビ(`.nav-bar`)でホーム/買い物/在庫を切り替える(`navigation.js`)。
